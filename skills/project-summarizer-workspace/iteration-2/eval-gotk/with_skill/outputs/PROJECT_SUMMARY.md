## Identity

- **What it is:** A tiny Go CLI ("Go Token Killer") that wraps high-bloat shell tools (`go test`, `git`, `grep`, `tsc`, `eslint`, `ls`, `tree`, `find`, `docker`, `kubectl`, `cargo`, `npm`, `curl`, plus a `read` subcommand for files) and reshapes their stdout into compact summaries before output reaches an LLM context window. Intentionally a stripped-down spiritual successor to [`rtk-ai/rtk`](https://github.com/rtk-ai/rtk).
- **Type:** CLI tool — specifically a **filter / wrapper / passthrough CLI** (it execs child processes, transforms stdout, forwards stderr, propagates exit code).
- **Status:** Pre-first-commit at time of summary — `git log` returns `fatal: your current branch 'main' does not have any commits yet` (verified via `git log --all --oneline`). Files are present in the working tree, version constant is `"0.1.0"` (`main.go:11`). README labels itself as v0.1.
- **Primary language(s):** Go 1.22 (`go.mod:3`).

## Conventions an agent must respect

Top rules; full set in `docs/project/conventions.md`.

- **Zero third-party Go dependencies.** Anything beyond the stdlib is a hard no — see README "Install" ("No third-party dependencies. `go.mod` has zero `require` lines"). Verified empty `require` block in `go.mod`. Adding a dep would also break the air-gapped install story.
- **Flat `package main` layout.** All `.go` files live at repo root in `package main` (verified: `head -1 *.go | sort -u` yields exactly `package main`). Do not introduce `cmd/`, `internal/`, or `pkg/` subdirs.
- **One filter per file, named `filter_<tool>.go`** with a sibling `filter_<tool>_test.go`. Exception: closely related tools share a file (`filter_listing.go` for `ls`/`tree`/`find`; `filter_misc.go` for the passthrough caps). Add new filters as `filter_<tool>.go` and register the handler in the `handlers` map at `main.go:14-29`.
- **Hand-rolled dispatcher.** No cobra/urfave-cli/etc. The map at `main.go:14-29` *is* the dispatcher. Don't introduce a CLI framework.
- **stdout is the only thing reshaped.** Stderr and exit code are forwarded verbatim. README: "Exit codes and stderr are preserved. Only stdout is reshaped." Verified: every filter sets `cmd.Stderr = os.Stderr` (exception: `filter_tsc.go:34` merges stderr into stdout — see Gotchas) and returns `execExitCode(...)`.
- **Filters degrade to passthrough on edge cases.** Watch mode, `--help`, `--version`, JSON-output flags, format-changing grep flags — each filter detects these and calls `passthrough(name, args)` (`common.go:13`) rather than parsing. Mirror this when adding a filter.
- **Tests use only `testing`** — no testify, no mocks. Table-driven style with `t.Run(name, func(t *testing.T))`. See `filter_git_test.go:TestHasFlag` as the canonical shape.

## Gotchas

- **No commits yet.** `git log` fails with `fatal: your current branch 'main' does not have any commits yet` (verified). Initial commit will need a `.gitignore` (currently exists at repo root — see `.gitignore` content) but no LICENSE has been chosen.
- **`tsc` filter merges stderr into stdout.** `filter_tsc.go:34` sets `cmd.Stderr = cmd.Stdout`. Unlike every other filter, `gotk tsc` does **not** preserve stderr verbatim. Intentional (TypeScript writes diagnostics to stdout in some configs) but worth noting if you debug missing stderr.
- **`gotk lint` forces `--format=json`** unless `--format`/`-f` is already in args (`filter_lint.go`). It calls `eslint` and parses JSON; if eslint exits ≥2 ("fatal" config error), it emits the raw stdout instead. If you wrap a non-ESLint linter through this, it will silently misbehave.
- **`gotk go ...` only filters `go test`.** `runGo` (`filter_gotest.go:1-20`) checks `args[0] == "test"`; everything else (`go build`, `go vet`, `go run`) is raw passthrough. So `gotk go build ./...` is identical to `go build ./...`.
- **`go test -json` is detected and bypassed.** `isJSONFlag` in `filter_gotest.go` falls through to `passthrough` so downstream JSON consumers (e.g. `gotestsum`) keep working.
- **`gotk git diff` line cap is 300.** Beyond that, you get `--stat` and a hint, *not* the diff (`filter_git.go:gitDiff`). To force-see a long diff, use `git --no-pager diff <path>` directly (the filter's own hint message says so).
- **`gotk read` defaults to 500 lines / 200 chars per line.** Pass `--all` for unlimited lines; `--raw` to drop to plain `cat`. Defined in `filter_read.go:readDefaultMaxLines` / `readDefaultMaxChars`.
- **History file write failures are silent.** `track.go:appendEvent` swallows all errors — if the home dir isn't writable or the path is invalid, you simply get no analytics, with no warning. Filters still run.
- **`cargo` is passthrough only** (300-line cap). The README ("When to use" table) explicitly notes: "v0.x doesn't parse `cargo test` fail blocks; full parser is a future iteration."
- **Hook-style invocation is *not* supported.** README "Non-goals": "Hooks. This tool is intentionally invocation-driven via model instructions". The expected pattern is to point an LLM at `GOTK.md` or `.github/copilot-instructions.md`, which tell the model to write `gotk <cmd>` instead of `<cmd>`. Don't try to `alias git=gotk git` — interactive `git commit`, `git rebase -i`, etc. would break (also called out in `GOTK.md` "When NOT to use").
- **No CI, no release pipeline.** `.github/workflows/` does not exist (verified). The only thing in `.github/` is `copilot-instructions.md`. Releases/tagging will need to be set up.
- **Distribution is build-from-source only.** README is explicit: "Designed for environments where you cannot install third-party binaries — you build it yourself from a stdlib-only Go source tree." A future maintainer adding a release-binary pipeline should not break the source-only install path.
