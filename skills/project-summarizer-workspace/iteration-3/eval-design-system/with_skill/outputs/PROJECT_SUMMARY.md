## Conventions an agent must respect

The top rules for safely modifying this code:

1. **All prop types live in `src/components.d.ts`** — a single 648-line file with 70 exported `interface`/`type` declarations. This is **intentional**, not technical debt. Add new prop interfaces there, not next to the component. (See "Don't refactor" below.)
2. **Re-export every new component from `src/index.tsx`.** Anything not re-exported is invisible to consumers — `src/index.tsx` is the contract.
3. **Style with `.rcs-*` class names defined in `src/styles.css`.** Don't introduce CSS-in-JS, CSS Modules, or Tailwind — the project deliberately uses hand-rolled BEM. 251 classes today (`grep -oE '\.rcs-[a-z0-9-]+' src/styles.css | sort -u | wc -l`).
4. **Token *types* go in `src/tokens.ts`; token *values* go in `colors_and_type.css`.** Components consume types at compile time and the CSS variables at runtime. Don't introduce a JS token object — there isn't one and there shouldn't be.
5. **Theme switching is `data-theme="light|dark"` on `<html>`.** Set by `ThemeProvider` (`src/components/theme.tsx:30–34`). Don't introduce alternative theming mechanisms (context-based, prop-drilled).
6. **Strong typing is the product.** From `src/components.d.ts:1–14`: discriminated unions for variants, strict event signatures (never `(value: any) => void`), `readonly` on array props. New code must match.
7. **Zero runtime deps.** Only `react` + `react-dom` as `peerDependencies` (`package.json:46–49`). Don't add a runtime dependency without explicit user approval.

## Gotchas

- **Build script concatenation, not bundling.** `pnpm build` in `package.json:37` runs `tsc` *and* a Node one-liner that does `writeFileSync('dist/styles.css', readFileSync('colors_and_type.css') + '\n' + readFileSync('src/styles.css'))`. If you rename either CSS file or move `colors_and_type.css` out of the project root, the build silently produces a broken stylesheet. Update the script in lockstep.
- **`tsconfig.build.json` excludes `src/styles.css` explicitly** (`tsconfig.build.json:8`). The build relies on this — TypeScript would otherwise try to type-check the CSS file via the broad `include` glob.
- **No ESLint config in the repo.** `package.json:39` declares `lint: eslint ...` and CI calls it, but no `.eslintrc*` / `eslint.config.*` exists (verify via `find . -maxdepth 2 -name '.eslintrc*' -o -name 'eslint.config.*'`). CI sets `continue-on-error: true` for the lint step (`.github/workflows/ci.yml:34`). Adding a config will turn lint failures into hard CI failures — coordinate the change.
- **No tests yet.** Vitest is wired but zero `*.test.*` / `*.spec.*` files exist (verify via `find . -name '*.test.*' -not -path '*/node_modules/*'`). CI also soft-fails the test step (`continue-on-error: true` at `.github/workflows/ci.yml:38`). When you add the first test, plan to remove that flag.
- **No Storybook or visual testing.** Component QA today is manual via the 38 HTML cards in `preview/` and the device-frame viewer at `preview/responsive-check.html`. Those cards are **static HTML** referencing the CSS — they do *not* import the React components. Visual changes must be checked manually in both `preview/` and the live components.
- **Release workflow refuses to publish if `package.json` version != git tag.** `.github/workflows/release.yml` reads version, compares with `${GITHUB_REF_NAME#refs/tags/}`, and `exit 1`s on mismatch. Use `pnpm version <bump>` (which writes both atomically) instead of editing `package.json` by hand.
- **`SKILL.md` at repo root is content shipped to package consumers**, not a project-summary doc. It's a Claude skill ("randomcodespace-design") that downstream prototypes invoke. Don't read it as authoritative project metadata.
- **`ui_kits/*/README.md` and the `preview/` HTML cards are reference *output*, not the library's source.** They contain inline React + Babel and use the CSS tokens directly. Editing them does not affect what the package publishes.
- **Branch is ahead of `origin/main` by 1 commit** (`git status`). The single non-initial commit is `checkpoint: pre-yolo` — likely a working snapshot, not a curated change. Don't treat that commit as the reference state.
